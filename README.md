# High-Frequency Event-Driven Backtesting Engine

An ultra-low-latency, event-driven quantitative trading backtester built in C++17, featuring a Python-based ETL data pipeline and risk analytics dashboard. 

This engine is designed to simulate high-frequency trading (HFT) and statistical arbitrage strategies with realistic market friction (slippage and commissions). It processes tick data in **sub-microsecond** latency (~725 nanoseconds per tick) by utilizing O(1) stateful indicators and fixed-point arithmetic for portfolio accounting.

## 🚀 Architecture

The system is decoupled into three distinct layers to maximize execution speed:

1. **ETL Data Pipeline (Python):** Ingests raw multi-asset market data from `yfinance`, cleans the data, forces chronological uniformity, and exports headerless CSVs to prevent branch-prediction misses in the C++ hot loop.
2. **Execution Engine (C++):** An event-driven state machine. It parses the pristine CSV, routes `MarketEvent`, `SignalEvent`, `OrderEvent`, and `FillEvent` objects, updates O(1) technical indicators, and maintains a multi-asset `Portfolio` ledger.
3. **Analytics & Risk (Python):** Parses the output `trades.csv` generated by the C++ engine to calculate institutional risk metrics (Sharpe Ratio, Maximum Drawdown) and render a Matplotlib equity curve.

## ⚙️ Features

* **Multi-Asset Routing:** Tracks state and accounting for a universe of interleaved symbols simultaneously via `std::unordered_map` referencing.
* **O(1) Technical Indicators:** Custom-built indicator library (SMA, EMA, RSI, Bollinger Bands) designed with sliding-window queues to prevent O(N) loop lookbacks.
* **Rigorous Unit Testing:** Mathematical validation of all quantitative indicators using Google Test (GTest) to prevent fixed-point truncation and floating-point drift.
* **Fixed-Point Accounting:** Protects portfolio cash balances from precision loss.
* **Realistic Execution Friction:** Simulates BPS-based slippage and flat-fee commissions.
* **Hardware Profiling:** Linux `perf` integration, Flamegraph generation, and built-in `std::chrono` high-resolution timers to profile the hot path.

## 🚦 Quick Start Guide

### 1. Setup the Python Environment
Create and activate a virtual environment, then install the required dependencies.
```bash
python3 -m venv venv
source venv/bin/activate
pip install pandas yfinance matplotlib numpy
```

### 2. Run the Data Downloader (ETL)
Pull the historical market data. By default, this downloads 60 days of 5-minute interval data and formats it for the C++ engine.
```bash
python download_data.py
```
*Expected Output: `historical_data.csv` (No headers, clean timeline).*

### 3. Compile the C++ Engine (Out-of-Source Build)
Use CMake to generate the build files and compile the system. You can choose between raw speed or hardware profiling mode.
```bash
mkdir build
cd build

# For Maximum Performance (Default: -O3 optimization)
cmake -DCMAKE_BUILD_TYPE=Release ..

# For Hardware Profiling & Flamegraphs (Includes debug symbols)
# cmake -DCMAKE_BUILD_TYPE=RelWithDebInfo ..

make
```

### 4. Run the Unit Tests
Verify the mathematical accuracy of the indicator library before executing a backtest.
```bash
./run_tests
```
*Expected Output: GTest suite passes flawlessly with green validation.*

### 5. Execute the Backtest
Run the compiled C++ binary. The engine will ingest the market data, execute the strategy, and output the trades.
```bash
./backtester
```
*Expected Output: Terminal printout of hardware execution speed and a `trades.csv` file generated in the root directory.*

### 6. Generate Risk Analytics
Navigate back to the root directory and run the visualization script to calculate the Sharpe Ratio, Maximum Drawdown, and generate the equity curve dashboard.
```bash
cd ..
python visualize.py
```
*Expected Output: Terminal printout of risk metrics and a `backtest_report.png` chart.*

## 🔬 Hardware Profiling & Engineering Insights

Using the Linux `perf` profiler and Brendan Gregg's Flamegraphs, the execution pipeline was analyzed at the silicon level to identify architectural bottlenecks. 

### The I/O and Parsing Tax
Flamegraph analysis revealed that the core event-driven state machine (`BacktestEngine`, `Strategy`, `Portfolio`) executes in a tiny fraction of total CPU time. The vast majority of execution time (~85-90%) is bottlenecked by the C++ Standard Library's CSV parsing—specifically `std::basic_istream` for file reading and `__GI____strtod_l_internal` for string-to-double conversion.
